01

Cloud Web Application

SpendOps Dashboard

Status: 要件定義・AWS構成図作成済み / 実装準備中

日常支出とAWS利用料金をまとめて可視化し、月別の支出傾向を分析するWebアプリです。PayPayはCSVアップロード、JCBカードと横浜銀行はGmail + GAS、AWS料金はCost Explorer APIで取り込み、DynamoDBに保存する構成を設計しています。

目的
単なる記録ツールではなく、準自動で集計し、月次分析・カテゴリ分析・改善提案を深く行える支出分析ダッシュボードを作ること。
構成
Next.js、Tailwind CSS、AWS Lambda、API Gateway、Cognito、DynamoDB、S3、GAS、Cost Explorer APIを想定。
工夫
PayPayのメール自動取得が安定しない前提を受け入れ、CSV取込履歴、未取込警告、重複判定、個人情報を保存しない設計に切り替えました。
担当範囲
作りたい機能、要件、AWS構成方針を自分で整理し、AIを壁打ち相手として文章化・構成図化しました。
公開予定
完成後に概要資料、構成図、必要に応じてTerraformコードを公開予定です。
概要資料を見る 構成図を開く

02

Serverless Automation

Gmail Priority Bot

Status: 要件定義・構成設計・Terraformでのリソース作成確認済み

就活・インターン関連のGmailを定期的に取得し、重要度とカテゴリを分類してGmailラベルを付与し、結果をダッシュボードで確認するMVPです。破壊的なメール操作を行わず、最小限のメタデータだけを扱う方針で設計しています。

目的
スカウト通知、レポート、広告などに埋もれやすい重要メールを、手作業に頼らず見つけやすくすること。
構成
EventBridge Scheduler、Lambda、Gmail API、DynamoDB、CloudWatch Logs、API Gateway、フロントエンドダッシュボード、Terraformを想定。
方針
機械学習やLLMは初期版では使わず、理由を説明できる決定的ルールで High / Medium / Low / Unknown に分類します。
担当範囲
要件定義、AWS構成、Terraformで作成するリソースの整理を担当。自分の意図をAIに伝え、文章と設計資料へ落とし込みました。
公開予定
面接時に必要に応じて概要資料、構成図、Terraformコードを提示できるよう準備中です。
概要資料を見る Terraformを見る
